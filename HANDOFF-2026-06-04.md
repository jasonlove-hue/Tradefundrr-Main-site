### Removed every effect on the options hero image (they softened it)

### THE actual "mouseover tilt" — a JS mouse-follow tilt script (root cause)
After all the above the image still tilted **on mouse-over, on a fresh PC** (ruling out cache). Cause: a dedicated inline `<script>` at the very bottom of `options.html`:
```js
/* Hero platform image: 3D mouse-follow tilt */
wrap.addEventListener('mousemove', e => img.style.transform='rotateY(...)rotateX(...)');
wrap.addEventListener('mouseleave', () => img.style.transform='rotateY(-10deg) rotateX(0deg)'); // resets to a STUCK -10deg
```
Flat on first load, but any mouse movement tilted it and `mouseleave` parked it at a permanent −10°. **Removed the script;** added **`transform:none!important`** to `.hero-platform-img` as a backstop (CSS `!important` beats a JS-set inline `style.transform`, which is normal priority).
**Debugging lessons:** a JS/mouse-driven transform is invisible to a resting `getComputedStyle` check and to a single static hover — reproduce it by dispatching a `mousemove` and reading `img.style.transform`, or inspect the live deploy with Chrome MCP `javascript_tool` (we used it on the live Vercel URL to read the computed transform, enumerate matched CSS rules, fire a synthetic mousemove, and prove it was JS, not CSS/cache). The bash-truncation issue above is why `grep` couldn't find the script for so long; `Read` surfaced it instantly.
