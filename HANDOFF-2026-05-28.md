## ⚠ Workspace Notes — Critical for Next Session

### Edit-tool size cap (~91 KB)

The harness's `Edit` tool silently truncates writes once a file passes approximately 91,000 bytes. The truncation is hard-capped to the same byte count regardless of how much content the Edit operation tries to add. Symptom: an Edit reports success, the size before and after is identical, but the file's closing tags (`</body></html></script>`) are missing.

**Workaround:** for any write touching a >80 KB file, use `python3` heredoc via `mcp__workspace__bash`:

```bash
python3 << 'PYEOF'
from pathlib import Path
P = Path("/sessions/zen-keen-pascal/mnt/TradeFundrr Brain/main-site/<file>")
src = P.read_text(encoding="utf-8")
# ... mutations ...
P.write_text(src, encoding="utf-8")
PYEOF
```

`bash` writes do NOT hit the cap. The Edit tool remains fine for small (<50 KB) files.

### Workspace mount cache lag

When a file is restored/modified on the user's Windows side (via OneDrive Version History or a Notepad save), the workspace mount on the sandbox side does NOT immediately see the new content. Cache invalidation can take 1-2 minutes or longer, and sometimes requires the user to rename the file or copy-and-rename to force a fresh sync.

**Workaround:** if you make Windows-side changes (or restore a previous version), have Jason create a NEW filename via copy + rename in File Explorer. The new filename forces OneDrive to treat it as a fresh upload, and the workspace mount sees it within ~30s.

This is what produced `stocks-preview-v2.html` — Jason copied the broken `stocks-page-mockup-preview.html` to a new filename so we could resume work on a clean mount-side view.

### File-integrity verification protocol

After every bash write to a critical file, verify:
1. File size grew (or shrank by an expected amount)
2. Closing tags are present: `grep -c "</body>" "$F"` and `grep -c "</script>" "$F"` should both return 1
3. `tail -3 "$F"` shows `</body>\n</html>\n` (or whatever the natural end-of-file should be)

If any of these fail, the write was truncated. Recover via OneDrive Version History on Jason's side and re-attempt via bash.

### Bash-driven file writes pattern

After the size-cap discovery, every substantial mutation moved to a `python3` heredoc:

```bash
python3 << 'PYEOF'
from pathlib import Path
P = Path("...")
src = P.read_text(encoding="utf-8")

# replace anchored text
OLD = "..."
NEW = "..."
assert OLD in src, "anchor missing"
src = src.replace(OLD, NEW)

P.write_text(src, encoding="utf-8")
print(f"OK, size: {len(src):,} bytes")
PYEOF
```

This pattern is repeatable and safe. It also makes the change atomic (full file read → mutate → full file write) which avoids partial-write states.
