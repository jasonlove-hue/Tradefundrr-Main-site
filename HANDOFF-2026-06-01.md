## ⚠ Workspace Notes — Critical for Next Session (carried + reinforced)

### The agent CANNOT git push
The sandbox has no GitHub credentials (`fatal: could not read Username`). **Jason pushes from his own machine.** After each change the agent provides the `git add / commit / push origin main` command; nothing is live until Jason runs it and Vercel rebuilds (~30–60s). Several "this didn't push / didn't change" moments this session traced back to either an un-run push **or** edits made only to a mockup file that were never ported into the live `pages/stocks.html`. **Always confirm: did the change land in the LIVE file, and was it pushed?**

### Mockup ≠ live
Review scratchpads (`*-mockup.html`, `*-preview.html`, `stocks-hero-*.html`) live in `main-site/` root and deploy at their own URLs. They are NOT the live page. Approved content must be hand-ported into `pages/stocks.html`. Don't assume editing a mockup updated the live page.

### Bash mount serves STALE / TRUNCATED copies of large HTML
Carried from prior handoffs and confirmed again: do **not** read/write the big HTML files through bash/cat/python/sed — use the `Read` / `Write` / `Edit` file tools, which see the true file. `git log/diff/status`, `grep`, and binary `cp` via bash are fine; reading/writing OneDrive-synced HTML content via bash is not. Verify content edits with `Grep` afterward.

### Cache / deploy
`/assets` is cached immutable, so CSS/asset changes need a `?v=` cache-buster bump. Inline HTML/CSS deploys fresh. After a push, hard-refresh (Ctrl+Shift+R) to beat the browser cache.

### OneDrive + git lock conflicts
Repo is in a OneDrive-synced folder; OneDrive re-locks `.git/*.lock`. If `git add`/`commit` breaks: pause OneDrive, `del /s /q ".git\*.lock"`, retry. Long-term fix: move the repo out of OneDrive.

### Same-origin iframe scroll-trigger pattern (new this session)
To make an embedded iframe's animation start on scroll-into-view *without parent edits*: freeze all CSS animations with `body:not(.go) * { animation-play-state: paused !important }`, then inside the iframe use `new window.parent.IntersectionObserver(...)` observing `window.frameElement` (same-origin) to add `.go` when it enters the parent viewport. `animation-play-state` freezes the whole timeline including delays, so staggered sequences resume cleanly from frame 0. Reusable for any future embedded-graphic animation.
