#### A.3 Pro-funding hero GIF → MP4/WebM video

The biggest single perf win of the day. The hero was loading a **5,066 KB animated GIF** from the old WordPress site. PageSpeed flagged: "Using video formats instead of GIFs could save 4,812 KiB."

Solution: downloaded the GIF locally (Jason ran `Invoke-WebRequest` since sandbox can't reach the WP host), converted with ffmpeg:

```bash
# MP4 (H.264) — universal browser support
ffmpeg -i futures_pro_animation_v2.gif -movflags +faststart \
       -pix_fmt yuv420p -c:v libx264 -preset slow -crf 23 \
       futures_pro_animation_v2.mp4
# Result: 5066 KB → 960 KB (-81%)

# WebM (VP9) — modern browsers, even smaller
ffmpeg -i futures_pro_animation_v2.gif -c:v libvpx-vp9 -crf 33 -b:v 0 \
       -row-mt 1 -pix_fmt yuv420p futures_pro_animation_v2.webm
# Result: 5066 KB → 495 KB (-90%)

# Poster (first frame as WebP)
ffmpeg -i futures_pro_animation_v2.gif -vframes 1 \
       futures_pro_animation_v2-poster.webp
# Result: 1.1 KB
```

Replaced the `<img>` with a `<video>` element offering both sources + poster + GIF fallback:

```html
<video class="pf-hero-video"
       autoplay loop muted playsinline preload="auto"
       fetchpriority="high"
       width="1000" height="563"
       poster="../assets/images/futures_pro_animation_v2-poster.webp?v=…"
       aria-label="Pro Trader Funding setup animation">
  <source src="../assets/images/futures_pro_animation_v2.webm?v=…" type="video/webm">
  <source src="../assets/images/futures_pro_animation_v2.mp4?v=…"  type="video/mp4">
  <img src="../assets/images/futures_pro_animation_v2.gif?v=…" alt="…" loading="eager" width="1000" height="563">
</video>
```

**Why this pattern matters:** any GIF-as-animation on the site should follow this. The `<video autoplay loop muted playsinline>` element is visually identical to a GIF but typically 10× smaller. `playsinline` is critical for iOS Safari (otherwise it tries to go fullscreen). `muted` is required for autoplay in most browsers.

## New Patterns Established (use these for future work)

These are reusable across the site. Future Claude instances: this is your toolkit.

### Pattern 1: Atomic Python writes for OneDrive resilience

The repo lives in OneDrive which truncates large files mid-write on save conflicts. **Never use multi-step Write/Edit for substantial changes.** Always use a single Python script that:
1. Reads the file
2. Performs all replacements via `str.replace()` with assertion-checked unique anchors
3. Verifies tag balance (`html.count("<section") == html.count("</section>")`)
4. Verifies `html.rstrip().endswith("</html>")`
5. Atomic-writes via `tempfile.mkstemp(dir=dirpath)` + `os.replace()`

This pattern is used in nearly every edit this session. Look at `/tmp/*.py` scripts in the conversation history for templates.

### Pattern 2: Image optimization workflow

For any image > 50 KB:
1. Check dimensions: PIL/Pillow `Image.open(path).size`
2. If displayed at less than half native size → resize with `Image.LANCZOS` to ~2× display size
3. Convert PNG → WebP: `im.save("file.webp", "webp", quality=85, method=6)` (or `lossless=True` for transparency)
4. Convert WebP at lower quality (78) for further savings on existing WebPs
5. Update all HTML refs with grep+sweep (use `?v=YYYYMMDDHHMM` cache buster)

### Pattern 3: GIF → MP4/WebM for animations

ALWAYS convert GIFs > 500 KB. The `<video autoplay loop muted playsinline>` element is visually identical to a GIF at 10× smaller size. Required attributes:
- `autoplay loop muted` — replicate GIF behavior
- `playsinline` — critical for iOS Safari (otherwise fullscreen)
- `preload="auto"` — fast playback start
- `width` + `height` — prevent CLS
- `fetchpriority="high"` — if it's the LCP
- Poster as WebP first frame — instant first paint while video loads
- Fallback `<img>` inside `<video>` for ancient browsers

### Pattern 4: Third-party lazy-load via IntersectionObserver

For any third-party script (HubSpot, Calendly, Tally, Trustpilot full widget, etc.):
1. Remove eager `<script>` from `<head>`
2. Add a placeholder `<div>` where the widget will mount, with `data-*` attributes for config
3. Add inline IO-based loader script that watches the placeholder
4. On viewport entry (typically `rootMargin: '400-600px 0px'`), inject the third-party `<script>` into `<head>`
5. In script.onload callback, initialize the widget targeted at the placeholder
6. Add `<link rel="preconnect">` for the third-party domain so DNS+TLS is warm

See pro-funding.html HubSpot implementation as the reference.

### Pattern 5: Critical font preloading

Any font weight used above the fold needs `<link rel="preload" as="font" type="font/woff2" crossorigin href="...">` in the `<head>` BEFORE the CSS link. Otherwise the browser only discovers the font after parsing CSS (chained request = slow LCP).

Default preloads on most pages: inter-400, inter-700. Pro-funding adds inter-500, inter-600, geist-mono-400 because those are LCP-critical there.

### Pattern 6: "Which means:" plain-English closer

Used in FAQs and the new Account Details accordion. Each technical answer ends with a `<strong class="stx-faq-means">Which means: [plain English]</strong>` line. Translates jargon for low-awareness visitors. Pattern is in `stocks-redesign-preview.html` and `pages/stocks.html` — search for `stx-faq-means`.

### Pattern 7: Review banner pattern (for in-progress design work)

When drafting multiple new sections for Jason to evaluate flow, wrap each in a visible yellow banner so he can spot every change while scrolling:

```html
<div class="stx-review-flag">
  <span class="stx-review-flag__tag">📝 NEW SECTION X</span>
  <span class="stx-review-flag__desc">Description of what this is + where it sits</span>
</div>
<section>...new content...</section>
```

After approval, strip all `.stx-review-flag*` and `.stx-review-banner-top` elements + the related CSS block in one pass. See `lowaware_patch.py` and `preview_final.py` in the conversation as references.

### Pattern 8: "Real X. Not Y." brand voice

Brand voice for contrasting our offering against alternatives. Use this pattern instead of "More X than Y" which reads awkwardly. Matches "Sim is sim. Live is live." Examples:
- Real rules. Not vague promises.
- Real fills. Not retail PFOF.
- Real desk capital. Not a bigger sim.

### Copy rule — NO EM DASHES in visible body text (Jason rule, set 2026-05-25)

Em dashes (`—` or `&mdash;`) are **banned from visible body copy on the site**. This applies to anything new you write — preview or production. Use **commas, periods, or colons** depending on context:

- "X — Y" where Y continues the clause → **comma**: "X, Y"
- "X — Y" where Y is a new sentence → **period**: "X. Y" (capitalize Y)
- "X — Y — Z" parenthetical → **commas**: "X, Y, Z"
- "X — explanation list" → **colon**: "X: explanation list"

**Exempt zones (em dashes allowed):**
- HTML comments (`<!-- … -->`) — never renders
- CSS comments (`/* … */`) — never renders
- `<script>` blocks — JavaScript syntax
- The footer + legal disclosure block (Jason explicitly excluded these from the sweep on 2026-05-25)

If you're adding a new section, run a `grep -c '—\|&mdash;'` on your new content before saving. Should be 0 in any visible-text region.

Pattern for cleaning up if you slip up:
```python
# In a Python script, after writing visible body content:
import re
content = re.sub(r'\s*(?:&mdash;|—)\s*(\S?)', lambda m: (". " if m.group(1).isupper() else ", ") + m.group(1), content)
```
